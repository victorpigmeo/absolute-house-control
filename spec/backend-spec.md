# Backend Spec

## Tech stack
- Language/framework: Java 25, Spring Boot 4.1.0
- Build tool: Gradle
- Database: PostgreSQL
- Auth provider: Keycloak
- Architecture style: microservices
- Deployment target: Kubernetes (Talos OS)
- Test framework: JUnit 5, AssertJ, Mockito (via `spring-boot-starter-test`), Testcontainers for integration tests
- Static analysis / lint: Spotless (formatting) + Error Prone (compile-time bug detection)
- Migration tool: Flyway
- API documentation: springdoc-openapi (`springdoc-openapi-starter-webmvc-ui`) — each service serves its OpenAPI spec at `/v3/api-docs`; new services should add the same dependency, since this is what [frontend-spec.md](frontend-spec.md)'s OpenAPI codegen step generates a typed client against.

## Confirmed decisions
- **Service communication**: plain REST for synchronous service-to-service and client-facing calls is the only confirmed communication style for now. An event-driven (message broker) style for asynchronous events/state changes is a possible future addition, not a current pattern — see [Open / not yet defined](#open--not-yet-defined); nothing today has a concrete async use case for it.
- **Gradle module layout**: a single multi-module Gradle build under `backend/` (one root `settings.gradle.kts`), with each microservice as its own subproject under `backend/services/<name>/`. Each service subproject still produces its own independently deployable artifact, built into a container image via Spring Boot's Buildpacks support (`bootBuildImage`) rather than separate per-service Gradle builds/repos. There's no shared `backend/common/` module for now — it was removed since nothing needs cross-service shared code yet; add one if/when a real need shows up, with an explicit statement of what belongs in it. Java package group id convention: `dev.pigmeo.ahc.<service>` (`ahc` = absolute house control) — new services should follow it.
- **Concrete domain microservices**: starting with two — `greenhouse` (manages everything related to the greenhouse) and `lighting` (manages everything related to house lights). More services may be added later as scope grows.
- **Per-service architecture**: each microservice uses a layered package structure — `application` (controllers, exception advice, message consumers, DTOs, mappers), `domain` (domain services, entities, exceptions), and `infrastructure` (configuration, repositories, message producers). All inbound input is mapped before entering the domain layer: state-changing/write input (e.g. an actuator command) becomes a command object carrying intent (e.g. `RunPumpCommand`), while read input maps directly to/from domain entities. All domain entities are mapped to response DTOs before being returned over HTTP.
- **Database strategy**: a single shared PostgreSQL instance with schema-per-service isolation — each microservice (e.g. `greenhouse`, `lighting`) gets its own schema and its own DB user/credentials scoped to only that schema (least privilege), rather than a separate Postgres instance per service. Keycloak shares this same instance, with its own schema and least-privilege user, rather than a second Postgres deployment (see [spec/infra-spec.md](infra-spec.md)'s Operators decision).
- **API gateway approach**: no dedicated app-level gateway service (e.g. Spring Cloud Gateway) — routing/exposure is handled per [spec/infra-spec.md](infra-spec.md)'s confirmed ingress mechanism, and each microservice validates Keycloak-issued JWTs itself via Spring Security's OAuth2 resource server. **Current code status**: this JWT validation is temporarily disabled in `greenhouse` (commit `ab1f1cb`) — `SecurityConfig` and the OAuth2 resource server dependency were removed so every endpoint is currently open; re-enablement is tracked as issue #40 (refs #41). See also [Open / not yet defined](#open--not-yet-defined).
- **Service discovery / configuration management**: no Spring Cloud infrastructure (Eureka/Consul, Config Server) — service discovery uses Kubernetes' built-in DNS (services call each other by their stable K8s Service DNS name), and configuration uses Kubernetes ConfigMaps (non-sensitive) and Secrets (DB credentials, Keycloak client secrets, broker credentials), consumed via Spring Boot's standard externalized configuration (env vars / mounted files). Config changes are applied via redeploy rather than dynamic runtime refresh — see [spec/infra-spec.md](infra-spec.md) for how that redeploy is actually triggered.
- **Keycloak realm/client structure**: a single realm, `absolute-house-control`. The frontend uses one public client, `web-ui` (Authorization Code + PKCE flow). Each microservice (`greenhouse`, `lighting`) is its own confidential client, acting as a resource server (validates JWTs via the realm's JWKS); synchronous service-to-service REST calls made on behalf of a logged-in user relay the original bearer token rather than minting a new one. No per-service machine-to-machine service account is provisioned for now — nothing has a no-user-context use case yet; add one when a real scenario needs it.
- **Greenhouse device integration (ESP32) — v1 contract**: a single ESP32 board exposes a small HTTP server controlling all three actuators and the DHT22 sensor; the `greenhouse` service is the only client, calling it directly over plain HTTP on the home LAN — no broker, no discovery protocol. For v1 the board's IP (`192.168.18.26`) is a fixed externalized config value — a Spring Boot `@ConfigurationProperties` value whose default comes from an env var populated from a Kubernetes ConfigMap (matching the `ESP32_BASE_URL` env var already used in `backend/README.md`) — not yet entered via a settings page or persisted in the database — see [Open / not yet defined](#open--not-yet-defined).
  - **Actuators** (LED, fan, pump) share one endpoint: `GET /api/gpio/set/{gpioPin}/{state}`, where `state` is `0` = ON, `1` = OFF (active-low). Pin mapping is fixed for v1: LED = `32`, pump = `25`, fan = `26`. The endpoint echoes what was applied, e.g. `{"pin": 26, "value": 0}`. The pump's run-for-duration behavior (FR3) is entirely backend-owned: the `greenhouse` service sends one ON command, schedules its own timer, and sends the OFF command itself when the duration elapses — the ESP32 only ever sees a plain on/off command, identical in shape to LED/fan. The backend enforces a hard maximum of 10 seconds on the requested duration (`RunPumpCommand.MAX_DURATION_SECONDS`) as a safety limit; requests above that are rejected the same way as invalid durations (FR4).
  - **Sensor**: `GET /api/dht` → `{"temperature": <°C>, "humidity": <%>, "heat_index": <°C>}`. `heat_index` is returned by the device but not required by any current functional requirement.
  - **Fire-and-forget, no state read-back (v1 limitation)**: the firmware has no endpoint to query an actuator's actual current state — `/api/gpio/set` is a one-way command. For v1 the `greenhouse` service therefore treats "the last command it successfully sent" as the device's current state, not a true hardware read-back. This last-known state is persisted to Postgres (per-device, in the `greenhouse` schema) rather than held in memory, so it survives service restarts and stays consistent if the service ever scales beyond one replica; the default before any command has been sent is OFF. This does not fully satisfy FR6 in `requirements/manage-greenhouse-devices/` ("reflect actual current state, not just the last command sent") and is a known, accepted v1 gap — closing it requires a firmware change to add a real state-query endpoint (see [Open / not yet defined](#open--not-yet-defined)).
  - **No backend-side polling loop for live reads**: the on-demand request path (what powers the live dashboard) never polls the ESP32 in the background — it calls `/api/dht` (and reports last-known actuator state) synchronously, only when it receives a request. See [frontend-spec.md](frontend-spec.md)'s v1 refresh model for how those requests get triggered. This is separate from the scheduled history-recording poller below, which runs independently of whether anyone has the dashboard open.
  - **Error handling deferred**: DHT read failures, invalid pin/state values, and the ESP32 being unreachable/timing out are explicitly out of scope for v1 — not handled, and not yet specified.
- **Greenhouse history recording**: a scheduled background job, separate from the on-demand live-read path above, polls the ESP32's `/api/dht` at least once every 60 seconds and persists each temperature/humidity reading (with its timestamp) to Postgres, regardless of whether the live dashboard is open. This is what satisfies `requirements/view-greenhouse-history`'s recording requirements, which the on-demand-only read path alone can't (it would only capture readings while someone happened to be viewing the dashboard).
- **ESP32 dev/test story**: automated tests run against a stub HTTP server (WireMock) simulating the ESP32's `/api/gpio/set` and `/api/dht` responses; manual local dev/verification targets the real board on the LAN at `192.168.18.26`. (The equivalent dev/test story for Tuya/`lighting` is still open — see below.)
- **Lighting device integration (Tuya)**: Tuya devices are controlled over Tuya's local LAN protocol rather than Tuya's Cloud API, to avoid a hard runtime dependency on Tuya's cloud (there is no official Java SDK for the local protocol — `com.tuya:tuya-spring-boot-starter` is Cloud-API-only and was ruled out for this reason). A dedicated sidecar (Python, wrapping the actively-maintained `tinytuya` library) owns local-protocol-version handling and on-LAN device discovery, deployed as its own Kubernetes Deployment/Service (not a same-pod sidecar container) reached over the cluster network; the `lighting` microservice talks to it over a plain internal REST API rather than implementing the protocol itself. That API's actual endpoint/payload shape and device-addressing scheme are deliberately left undefined for now — this architecture was locked in ahead of any functional lighting requirements because the local-vs-cloud protocol choice is a hard technical constraint worth settling early; the API contract itself should be designed once `requirements/` has a lighting requirement to shape it. Device `local_key`s are obtained out-of-band for now; cloud-based key bootstrapping is deferred.
- **Testing approach**: unit tests use JUnit 5/AssertJ/Mockito against the domain layer in isolation; integration tests use Testcontainers to run against a real Postgres instance (and, once chosen, Keycloak/the broker) rather than mocking infrastructure — consistent with this spec's general preference for testing against real dependencies. This split is enforced via two Gradle tasks: `test` (excludes `@Tag("integration")`, needs no Docker, part of the default `build`/`check`) and a custom `integrationTest` task (runs only `integration`-tagged tests, requires Docker, deliberately excluded from `build`/`check` so the default build stays usable in Docker-less environments) — `.github/workflows/backend-ci.yml` runs both as separate steps.
- **REST error response shape**: a project-wide `ErrorResponse(String error, List<String> details)` DTO, returned by a `@RestControllerAdvice` (`GreenhouseExceptionHandler`) for validation errors, malformed JSON, and domain exceptions (all as 400s). This is a real cross-service contract, not just a backend implementation detail: the frontend parses each `details` entry's `"field: message"` shape to populate per-field form errors (see [frontend-spec.md](frontend-spec.md)'s API client strategy) — any new validation error added here should keep that shape.
- **Spring profiles**: `local` (default, no `SPRING_PROFILES_ACTIVE` needed) and `prod`. `prod` has no defaults for `GREENHOUSE_DB_URL`, `GREENHOUSE_DB_PASSWORD`, or `KEYCLOAK_ISSUER_URI` — all three must be supplied via env vars (e.g. Kubernetes ConfigMap/Secret-mounted), or the application fails fast at startup ("could not resolve placeholder") rather than silently falling back to a local-dev credential.
- **Light-cycle scheduling**: a real subsystem implemented in `greenhouse` for `requirements/schedule-greenhouse-led-cycles/` — a `LightCycle` entity/migration (`V3__light_cycle.sql`) and a create endpoint (`POST /api/greenhouse/light-cycles`), validating its two cron expressions via a custom `@ValidCron` Bean Validation annotation (delegates to Spring's `CronExpression.isValidExpression`). Only creation (US1) is built so far — list/activate/automatic-scheduling-with-restart-resume (US2–US4) are not yet implemented, see [Open / not yet defined](#open--not-yet-defined).
- **Static analysis / lint tooling**: Spotless (e.g. `google-java-format`) for auto-fixable formatting, plus Error Prone for compile-time bug-pattern detection. Checkstyle/PMD and SonarQube were considered and skipped — they add more rule-curation overhead (PMD/Checkstyle) or require a running server (SonarQube) than this lightweight pairing, for similar payoff on a solo-maintained backend.
- **Migrations**: Flyway, run per-service against its own schema — each microservice owns its own migration history table and `db/migration` script set within its own schema, consistent with the schema-per-service/least-privilege database strategy above. Migrations run via Spring Boot's default Flyway auto-run on startup; Flyway's own advisory locking safely serializes concurrent migration attempts if a service ever runs with multiple replicas during a rolling deploy, so this isn't a gap to revisit later.
- **Local dev loop**: Postgres and Keycloak run as local containers during development, with the Spring Boot services connecting to them directly (rather than via Testcontainers-managed dev-mode). The message broker isn't part of this yet since it isn't chosen.

## Open / not yet defined
- Message broker / event-driven communication (deliberately deferred until a concrete async use case requires one — REST is the only confirmed communication style until then)
- Pump auto-shutoff durability: the pump's run-for-duration timer (FR3) is scheduled in-process by the `greenhouse` service. If the pod restarts mid-run, nothing currently resumes or forces the OFF command, which could leave a physical water pump running. Accepted as a known v1 limitation rather than fixed now; revisit (e.g. persisting the pending shutoff deadline so a restarted pod can recover it) if this actually bites.
- Tuya `local_key` provisioning automation (e.g. a one-time Tuya Cloud API bootstrap step) — deferred; keys are obtained out-of-band for now.
- Local dev story for Tuya/`lighting` — e.g. whether to stub/simulate devices for development away from the physical hardware, or always develop on the home LAN with real hardware reachable — deferred. (The ESP32/`greenhouse` equivalent is resolved — see "ESP32 dev/test story" above.)
- ESP32 firmware update to add a real GPIO state read-back endpoint (today's `/api/gpio/set` is fire-and-forget, command-only) — deferred; tracked as a future infra task to update the ESP32 code, to be filed once picked up.
- Error handling for the ESP32 HTTP calls (DHT read failures, invalid pin/state values, device unreachable/timeout) — deferred for v1; needs definition before this integration is considered production-hardened.
- Settings-page-driven, DB-persisted ESP32 IP/pin configuration — deferred; v1 hardcodes the IP (`192.168.18.26`) and pin mapping (LED `32`, pump `25`, fan `26`) as fixed config.
- Tuya sidecar's internal REST API contract (endpoints, payloads, device addressing) — deferred until lighting functional requirements exist to shape it.
- Keycloak/JWT validation re-enablement in `greenhouse` — disabled since commit `ab1f1cb` so every endpoint is currently open; despite being a "Confirmed decision" above, it isn't implemented right now. Tracked as issue #40 (refs #41).
- Light-cycle scheduling US2–US4: list all light cycles, activate a cycle (with switch confirmation), and automatic LED on/off control by the active cycle's cron schedule (including resuming after a service restart) — only US1 (create) is implemented today. See `requirements/schedule-greenhouse-led-cycles/`.
