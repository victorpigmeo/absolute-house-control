# Infra Spec

## Tech stack
- Orchestration: Kubernetes
- Node OS: Talos OS
- Target environment: home-lab bare-metal (physical/on-prem nodes)

## Confirmed decisions
- **Cluster topology**: 3 Talos VMs on the user's Proxmox host, each running combined control-plane + worker roles (Talos supports this directly via machine-config). 3 is the etcd quorum sweet spot (not 1, not an even number) — it survives any single node being down or mid-upgrade without losing quorum, enabling zero-downtime rolling Talos/Kubernetes upgrades one node at a time. At this workload's actual size (2 small microservices + a sidecar, Postgres, Keycloak, the frontend, eventually one broker) dedicated worker-only VMs would add no real isolation benefit, since all VMs share the same underlying host's resources anyway. Note: since all 3 VMs run on one physical Proxmox host, this is **not** hardware-failure HA — if the host dies, the whole cluster goes down regardless of node count. The 3-node shape is chosen anyway because this is a study project aimed at replicating a real-world production topology, not just the minimum needed to run a home greenhouse/lighting setup; what it actually buys here is OS/software-level fault isolation and safe rolling upgrades, not hardware fault tolerance. More worker capacity can be added later as a new VM if resource pressure shows up.

- **Networking/CNI choice**: Cilium (eBPF-based), replacing Talos's default Flannel and kube-proxy. Talos ships Flannel by default if untouched — fine for basic pod networking, but no NetworkPolicy support and not representative of how real clusters are built today, which matters given this is a study project aimed at replicating real-world patterns. Cilium is the eBPF-based CNI that's increasingly the production standard (underlies GKE's Dataplane V2, CNCF graduated project), and Talos has first-class documented support for installing it. Calico was the other genuine contender (also production-grade, solid NetworkPolicy support), but Cilium's Hubble observability — a UI/CLI visualizing live service-to-service traffic, DNS lookups, and policy verdicts — has direct learning value: you get to watch traffic flow between `greenhouse`/`lighting`/Postgres/Keycloak, not just configure it blind. Install mechanism: boot Talos with its built-in CNI disabled (`cni.name: none`), bootstrap the cluster, then install Cilium via Helm as the first workload — Talos's standard "bring your own CNI" pattern. Also enabling Cilium's eBPF kube-proxy replacement (with Talos's own kube-proxy disabled via `proxy.disabled: true`) and Hubble (Relay + UI), both effectively free once Cilium is in. Left for later (tracked in a provisioning task): actual NetworkPolicy authoring strategy (e.g. default-deny per namespace) and specific pod/service CIDR ranges — this entry just establishes Cilium as the capability to use when those are decided.

- **Ingress controller and TLS strategy**: Cilium's own Gateway API implementation (`gatewayAPI.enabled=true`) handles ingress — not a separate controller. The historical default, ingress-nginx, was officially retired (repo archived March 2026, no further releases or security fixes), ruling it out for a new project; Gateway API is the actively-maintained direction, and Cilium's implementation is fully conformant and already sits in the data path (no second L7 proxy to run), building directly on the `kubeProxyReplacement=true` already enabled in the CNI decision. Only the frontend and Keycloak get exposed Gateway `HTTPRoute`s/hostnames on the LAN (e.g. `house.yourdomain.com`, `auth.yourdomain.com`) — Keycloak needs its own route because the browser is redirected there directly during the OIDC Authorization Code flow. `greenhouse`/`lighting` get no external route at all, consistent with the frontend's API client strategy (reached in-cluster only). TLS comes from cert-manager using the ACME DNS-01 challenge against Cloudflare: the domain (registered at Squarespace, which has no cert-manager DNS-01 integration) keeps its registration there but has its nameservers pointed at Cloudflare (Squarespace supports custom nameservers), and cert-manager's native Cloudflare solver issues real publicly-trusted Let's Encrypt certs via DNS TXT records — no port-forwarding or public exposure needed for the challenge itself. The Cloudflare API token is scoped to just that one zone (least privilege) and stored as a Kubernetes Secret — the storage mechanism depends on the still-open secrets management approach below.

- **Secrets management approach**: SOPS encrypted with `age`, secrets committed encrypted into the GitOps repo — not Vault and not External Secrets Operator. Vault is the most "textbook" real-world secret manager, but self-hosting it adds genuine solo-maintainer tax (unsealing after every restart, or an auto-unseal path that typically needs a cloud KMS, which doesn't fit a home-lab/on-prem project, plus backing up unseal keys and running yet another stateful service) for no real benefit, since nothing here needs Vault's actual differentiators (dynamic/leased credentials) — the backend already chose static per-service DB users. External Secrets Operator is also a genuine production pattern, but it needs an external secret-store backend to sync from, and every realistic option (cloud secret managers, hosted SaaS vaults) pulls in a dependency that doesn't fit a project whose whole infra premise is on-prem with no cloud account. SOPS + age needs nothing beyond local tooling and git: secrets are authored locally as plain Kubernetes `Secret` manifests, encrypted with `sops` (age backend — a small keypair, no PGP web-of-trust/agent complexity) before being committed, and decrypted in-cluster by whichever GitOps controller is chosen later (Flux has native SOPS decryption built in; ArgoCD needs the KSOPS plugin) — see the Deployment/GitOps approach below for which one. This is a genuinely common real-world pattern in its own right (how a large share of FluxCD users handle secrets), not a home-lab shortcut. The one thing that can never live in git under this scheme is the age private key itself — every approach has some version of this bootstrap problem (Vault has unseal keys/root tokens) — it gets created once, injected into the cluster out-of-band, and backed up outside the cluster (e.g. a password manager). This covers the Cloudflare API token flagged in the ingress/TLS decision, plus the rest of the secret material already named across the spec: per-service Postgres credentials, Keycloak confidential-client secrets, and (once chosen) broker credentials.

- **Deployment/GitOps approach**: FluxCD (Flux v2), single-cluster, pull-based, with image automation against GHCR — chosen over ArgoCD because the secrets decision above already leans on Flux's native SOPS decryption (no extra plugin, vs. ArgoCD's KSOPS), and Flux is just as legitimately "real-world production" as Argo (it's where the term GitOps originated, and it underlies managed integrations like Azure Arc's GitOps feature). Argo's main edge is its UI, a nice-to-have for a solo operator rather than a functional need, and running it adds its own ingress/auth/RBAC surface to maintain; Flux is also composed of small single-purpose controllers rather than one monolithic app, consistent with this spec's recurring preference for fewer moving parts (skipping Spring Cloud infra, a dedicated API gateway, and Vault). Repo layout, Flux's own canonical shape, under `infra/`: `infra/clusters/home/flux-system/` (bootstrap-managed, not hand-edited), `infrastructure/` (Cilium, cert-manager, Postgres, Keycloak, broker), `apps/` (`greenhouse`, `lighting`, frontend) — `infrastructure/` reconciles before `apps/` via Flux `Kustomization` dependency ordering, since the apps need Postgres/Keycloak/cert-manager up first. Reconciliation is pull-based only: Flux's source-controller polls/webhooks the repo and kustomize-controller/helm-controller continuously reconcile the cluster to match it — nothing (no CI step, no human) ever runs `kubectl apply` against the cluster directly, which is the actual substance of "GitOps" here. Image promotion uses Flux's image-reflector/image-automation controllers, watching for new tags from the backend's already-chosen Buildpacks builds and auto-committing the version bump back into git, closing the CI→CD loop without CI touching the cluster or hand-edited manifests; images are pushed to GitHub Container Registry (`ghcr.io`) since it's free and already authenticated through the same GitHub account. Bootstrap is a one-time `flux bootstrap github` run that installs Flux and writes its own manifests into the repo — the one "ignition" step that isn't itself GitOps, the same shape as the age-key bootstrap problem above. Single cluster, no environment promotion pipeline — just one Flux `Kustomization` reconciling one directory.

- **Persistent storage approach**: Longhorn, replicated 3x across the Talos nodes — not local-path-provisioner, Rook-Ceph, or a Proxmox-native CSI driver. local-path-provisioner pins a volume to whichever single VM provisioned it, no replication, no failover — that would quietly undercut the entire point of the 3-node topology, since a node going down would just take the database with it. Rook-Ceph is the more "enterprise-grade" answer but is heavier operationally than a 3-node home-lab cluster proportionally needs — the same "real tool, disproportionate solo-maintainer tax" reasoning already applied to ruling out Vault and SonarQube. A Proxmox-native CSI driver doesn't add real node-failure resilience here either, since all 3 Talos VMs sit on one physical Proxmox host with no Proxmox-level clustering — Longhorn instead replicates at the Kubernetes/VM layer, so if one Talos VM goes down the other two still hold full replicas and the volume/pod can fail over without data loss, directly continuing the same "software/VM-level fault tolerance, not hardware-level" story already established for cluster topology. Replica count is 3, matching node count, for the same reasoning as the etcd quorum choice. Longhorn on Talos needs the `siderolabs/iscsi-tools` and `siderolabs/util-linux-tools` system extensions added to the Talos machine config (Longhorn depends on `iscsid`/`iscsiadm` plus `fstrim`), the `nbd`, `iscsi_tcp`, `iscsi_generic`, and `configfs` kernel modules, and extra kubelet data-path mounts — Talos-specific requirements, not optional. What actually needs persistent volumes here: Postgres (and whichever DB backs Keycloak) and, once chosen, the message broker — everything else (frontend, Cilium, cert-manager, Flux, the Tuya sidecar) is stateless or already stores its state in Postgres. Left open deliberately: where Longhorn's snapshot backups actually land (an S3-compatible target — a NAS, Backblaze B2, Cloudflare R2, etc.) — a real external choice, not something to guess at here.

- **How Keycloak, Postgres, and the message broker are deployed**: each runs via its own Kubernetes Operator rather than a bare Deployment/StatefulSet or generic Helm chart — operators encode the ongoing operational lifecycle (failover, backups, safe upgrades) as a controller that keeps running in-cluster, instead of "apply once and hope," which matters for stateful infrastructure like this. Postgres runs via the **CloudNativePG (CNPG)** operator (CNCF Sandbox, applying for Incubation, widely regarded as the standard way to run Postgres on Kubernetes) — one `Cluster` resource with 3 instances (1 primary + 2 standby, automatic failover via CNPG itself), continuing the same "3" HA pattern already used for the Talos nodes and Longhorn replicas, backed by Longhorn PVCs. It stays a single shared instance per backend-spec's schema-per-service strategy; Keycloak gets its own schema/least-privilege user on this same instance rather than a second Postgres deployment. Keycloak runs via the **official Keycloak Operator**, not a generic Helm chart — its `Keycloak` CRD manages the deployment (can scale to multiple replicas using Keycloak's built-in Infinispan clustering for HA), and its `KeycloakRealmImport` CRD handles the realm import declaratively, resolving the client secret via `spec.placeholders` from a Secret reference. That's the mechanism that closes out backend issue #17: once the hardcoded literal there is replaced with an env-var placeholder, the Operator resolves it from a SOPS-managed Secret at import time instead of a plaintext JSON file. The message broker (still backend-spec's choice to make) gets deployed via its own dedicated operator where one exists — e.g. Strimzi for Kafka, the RabbitMQ Cluster Operator for RabbitMQ — same operator-first pattern, with storage on Longhorn PVCs; concrete specifics stay deferred until that broker choice is made. Each gets its own namespace under `infra/clusters/home/infrastructure/` per the GitOps layout, reconciled before `apps/`.

- **External access (public domain exposure)**: Cloudflare Tunnel (`cloudflared`), not a router port-forward and not ngrok — the home router currently isn't accessible to configure a port-forward, and a tunnel needs no inbound port at all: `cloudflared` runs as a normal in-cluster Deployment and makes only outbound connections to Cloudflare's edge, which then proxies `house-control.pigmeo.dev` traffic back through that tunnel. This builds directly on the Cloudflare account/zone already committed to for cert-manager's DNS-01 challenge in the ingress/TLS decision above — one Cloudflare account covers both DNS and the tunnel, rather than introducing ngrok as a second unrelated third-party account/process. ngrok was the initial idea but was passed over: free-tier ngrok hands out its own random subdomain (a paid plan is needed for a stable custom domain) and runs as a separate agent/account with no tie to the domain's actual DNS, whereas Cloudflare Tunnel issues a stable hostname on the existing domain for free and creates its DNS record automatically (`cloudflared tunnel route dns`, a CNAME to `<tunnel-id>.cfargotunnel.com`). Tailscale Funnel was also considered — also no port-forward needed — but it's built around Tailscale's own identity/ACL model for exposing a node, a better fit for private admin access to the cluster than for serving a clean public hostname on an existing domain. `cloudflared`'s ingress rule targets the in-cluster Cilium Gateway (not the frontend Service directly), so the existing Gateway API/TLS path above stays the single entry point inside the cluster — the tunnel only solves how traffic reaches that Gateway without an open inbound port. The tunnel credentials (the JSON file from `cloudflared tunnel create`) are stored as a Kubernetes Secret per the SOPS+age secrets approach above, same as every other credential in this spec. This is a generally-applicable pattern, not just a stopgap for the missing port-forward — the home router's public IP is never directly exposed either way, so it can stay in place even after modem/port-forward access returns.
