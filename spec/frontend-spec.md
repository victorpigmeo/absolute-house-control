# Frontend Spec

## Tech stack
- Framework: Next.js
- Language: TypeScript, with strict typing enabled (`strict: true`)

## Confirmed decisions
- **Application/route structure**: Next.js App Router (not Pages Router) — the current Next.js default, and Server Components/Server Actions fit well with calling backend REST APIs and handling the Keycloak session server-side. Top-level structure mirrors the backend's domain microservices: a shared authenticated dashboard shell (`(dashboard)/layout.tsx`) with one route folder per domain (`greenhouse`, `lighting`, ...), each with its own `settings` page for domain-specific device config (e.g. ESP32 IPs for `greenhouse`, Tuya devices for `lighting`, per [spec/backend-spec.md](backend-spec.md)). New domains added later just add a new folder under the dashboard shell, matching how the backend adds a new `services/<name>/`. The entire app sits behind Keycloak auth except a `login` route handling the Authorization Code + PKCE redirect/callback.
- **Auth integration with Keycloak**: Auth.js (NextAuth.js v5) with its built-in Keycloak provider, configured against the `web-ui` public client (Authorization Code + PKCE, no client secret) — chosen over Keycloak's own `keycloak-js` adapter (deprecated upstream, browser-only token storage) and over hand-rolled OIDC (reinvents PKCE/state/cookie-crypto handling Auth.js already solves). JWT session strategy: Keycloak's access/refresh/id tokens live in an encrypted, httpOnly cookie — no session DB/Redis, consistent with the backend's "skip extra infra until needed" pattern. Token refresh happens in Auth.js's `jwt` callback (check expiry, call Keycloak's token endpoint with the `refresh_token`; on failure, drop the session and force re-login). Route wiring: `app/api/auth/[...nextauth]/route.ts` is Auth.js's catch-all handler; `app/login/page.tsx` is just a "Sign in" button calling `signIn("keycloak")`; `middleware.ts` redirects unauthenticated requests to `/login` for everything under `(dashboard)`, backed by a server-side `auth()` check in `(dashboard)/layout.tsx` as defense-in-depth (current Next.js guidance is not to rely on middleware alone for gating). Logout is RP-initiated — it redirects through Keycloak's `end_session_endpoint` rather than only clearing the local cookie, so the Keycloak SSO session actually ends. The access token used on calls to `greenhouse`/`lighting` comes from this server-side session (`auth()` in Server Components/Actions/Route Handlers); the fetch-wrapper mechanics themselves are covered by the API client strategy decision below.
- **API client strategy**: all calls to `greenhouse`/`lighting` happen server-side only (Server Components for reads, Server Actions for mutations, Route Handlers where needed) — never directly from browser JS. This keeps Keycloak access tokens server-side (consistent with the auth decision's httpOnly-cookie session) and means the microservices never need CORS config or public exposure; only the Next.js app itself needs to be externally reachable. Next.js reaches each service over its in-cluster Kubernetes Service DNS name (consistent with the backend's service-discovery decision), via a `GREENHOUSE_SERVICE_URL` / `LIGHTING_SERVICE_URL`-style env var per service. Clients are typed via OpenAPI codegen rather than hand-maintained fetch wrappers: each backend service emits an OpenAPI spec via `springdoc-openapi-starter-webmvc-ui`, and the frontend generates TS types plus a thin typed client per service with `openapi-typescript` + `openapi-fetch` (a small wrapper over native `fetch`, no heavy codegen runtime) — this avoids hand-mirroring Java DTOs in TypeScript, the main type-drift risk in a solo-maintained cross-language stack. Each generated client is wrapped with a thin layer that pulls the access token from the Auth.js session (`auth()`) and sets it as the `Authorization: Bearer` header. Reads default to `cache: 'no-store'` rather than Next.js's default cached `fetch`, since for live device/system state staleness is worse than the extra backend round-trip.
- **Real-time update mechanism**: Server-Sent Events, relayed through the Next.js server — not WebSockets, and not direct browser-to-backend connections. Each backend service exposes its own SSE endpoint for domain state-change events (sensor readings for `greenhouse`, on/off state for `lighting`) via Spring MVC's `SseEmitter` (no new dependency, no move to WebFlux); how each service gets fresh data internally (ESP32 polling, tinytuya status) stays that service's own concern per backend-spec.md. The browser never connects to a backend service directly: a Route Handler per domain (`app/api/stream/greenhouse/route.ts`, `app/api/stream/lighting/route.ts`, mirroring the existing per-domain route structure) checks the session cookie, attaches the Bearer token server-side, opens the SSE connection to the backend service, and pipes it to the browser, which just uses a plain same-origin `EventSource`. This isn't only for consistency with the API client strategy above — browser `EventSource` can't set an `Authorization` header at all, so a direct browser→backend stream would need the token in a query string (logged, leaked); relaying through Next.js avoids that outright. SSE was chosen over WebSockets because commands already go through normal request/response Server Actions, so there's no need for client→server push on this channel — SSE is plain HTTP, gets free reconnection from `EventSource`, and avoids WebSockets' ingress/protocol-upgrade complexity for no real benefit here. Standard SSE keep-alive (periodic comment lines) is enough to hold connections open through proxies. Note for infra-spec.md's eventual ingress controller/TLS strategy: these routes are long-lived and need a correspondingly long proxy idle-timeout.
  - **v1 exception — greenhouse**: per [backend-spec.md](backend-spec.md)'s ESP32 v1 contract, the `greenhouse` service has no background polling loop and no actuator state read-back, so there's nothing yet for an SSE endpoint to push. The greenhouse domain therefore skips the SSE route handler for v1: a Client Component polls a normal Route Handler/Server Action on a fixed interval (proposed default 5s, matching the DHT22's own minimum sampling interval) to refresh the DHT reading and last-known actuator state, with the backend calling the ESP32 fresh on each of those requests. This is a deliberate, temporary deviation — the SSE relay above remains the target architecture for greenhouse once the ESP32 firmware update (deferred, see backend-spec.md's Open items) adds real state read-back.
- **Styling and component primitives**: Tailwind CSS — it's Next.js's own scaffolding default, has zero runtime cost (compiles to static CSS, nothing executes per-request), and is the only mainstream styling option with clean Server Components compatibility (CSS-in-JS libraries like styled-components/Emotion still need `"use client"` workarounds and lack first-class RSC support). Paired with shadcn/ui (Radix UI underneath) for interactive primitives (switches, sliders, dialogs, forms) — components are copy-pasted into the repo rather than pulled in as an opaque dependency, keeping full control consistent with this spec's general preference against heavyweight abstractions, while still getting accessibility (ARIA, keyboard nav) for free from Radix.
- **Testing frameworks**: Vitest over Jest for unit/component tests — esbuild/Vite-based so substantially faster, native ESM (no config fighting with the App Router), and close enough to Jest's API to have no real learning-curve cost; it's also what Next.js's own docs now point to for new projects. React Testing Library sits alongside it, scoped to Client Components, hooks, and pure functions — Server Components don't have a stable isolated-rendering story for RTL today, so route-level behavior involving them is verified through Playwright instead. Playwright over Cypress for E2E — genuinely multi-browser, faster, native TypeScript, and supports multiple isolated browser contexts in one test, which matters for the Keycloak Authorization Code redirect leaving and re-entering the app. Mirrors the backend's existing test philosophy (prefer testing against real dependencies, per backend-spec.md): unit/component tests mock the API layer (MSW, or stubbing the generated OpenAPI client) since they test units in isolation, but Playwright E2E runs against the real `greenhouse`/`lighting` services, Postgres, and Keycloak rather than mocked ones.
