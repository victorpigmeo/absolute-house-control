# Frontend Spec

## Tech stack
- Framework: Next.js
- Language: TypeScript, with strict typing enabled (`strict: true`)
- Package manager: npm
- Static analysis / lint: ESLint (`eslint-config-next` defaults)

## Confirmed decisions
- **Application/route structure**: Next.js App Router (not Pages Router) — the current Next.js default, and Server Components/Server Actions fit well with calling backend REST APIs and handling the Keycloak session server-side. Top-level structure mirrors the backend's domain microservices: a shared authenticated dashboard shell (`(dashboard)/layout.tsx`) with one route folder per domain (`greenhouse`, `lighting`, ...), each with its own `settings` page for domain-specific device config (e.g. ESP32 IPs for `greenhouse`, Tuya devices for `lighting`, per [spec/backend-spec.md](backend-spec.md)), plus a `(dashboard)/page.tsx` dashboard home page that's the default landing target after login when there's no originally-requested page to return to. New domains added later just add a new folder under the dashboard shell, matching how the backend adds a new `services/<name>/`. The entire app sits behind Keycloak auth except a `login` route; that route itself is just the sign-in entry point (see the auth integration decision below for where the actual Authorization Code + PKCE redirect/callback is handled).
- **Auth integration with Keycloak**: Auth.js (NextAuth.js v5) with its built-in Keycloak provider, configured against the `web-ui` public client (Authorization Code + PKCE, no client secret) — chosen over Keycloak's own `keycloak-js` adapter (deprecated upstream, browser-only token storage) and over hand-rolled OIDC (reinvents PKCE/state/cookie-crypto handling Auth.js already solves). JWT session strategy: Keycloak's access/refresh/id tokens live in an encrypted, httpOnly cookie — no session DB/Redis, consistent with the backend's "skip extra infra until needed" pattern. Token refresh happens proactively in Auth.js's `jwt` callback (refresh once the access token has under ~60s remaining, not only after it's already expired, so in-flight requests don't race an expiring token; call Keycloak's token endpoint with the `refresh_token`; on failure, drop the session and force re-login). Route wiring: `app/api/auth/[...nextauth]/route.ts` is Auth.js's catch-all handler and is what actually performs the Authorization Code + PKCE redirect/callback; `app/login/page.tsx` is just a "Sign in" button calling `signIn("keycloak")`; `middleware.ts` redirects unauthenticated requests to `/login` for everything under `(dashboard)`, backed by a server-side `auth()` check in `(dashboard)/layout.tsx` as defense-in-depth (current Next.js guidance is not to rely on middleware alone for gating). Logout is RP-initiated — it redirects through Keycloak's `end_session_endpoint` rather than only clearing the local cookie, so the Keycloak SSO session actually ends. Session lifetime itself isn't set here — it's governed by the `absolute-house-control` Keycloak realm's SSO session settings (realm import config, not this app). The access token used on calls to `greenhouse`/`lighting` comes from this server-side session (`auth()` in Server Components/Actions/Route Handlers); the fetch-wrapper mechanics themselves are covered by the API client strategy decision below.
- **API client strategy**: all calls to `greenhouse`/`lighting` happen server-side only (Server Components for reads, Server Actions for mutations, Route Handlers where needed) — never directly from browser JS. This keeps Keycloak access tokens server-side (consistent with the auth decision's httpOnly-cookie session) and means the microservices never need CORS config or public exposure; only the Next.js app itself needs to be externally reachable. Next.js reaches each service over its in-cluster Kubernetes Service DNS name (consistent with the backend's service-discovery decision), via a `GREENHOUSE_SERVICE_URL` / `LIGHTING_SERVICE_URL`-style env var per service. Clients are typed via OpenAPI codegen rather than hand-maintained fetch wrappers: each backend service emits an OpenAPI spec via `springdoc-openapi-starter-webmvc-ui`, and the frontend generates TS types plus a thin typed client per service with `openapi-typescript` + `openapi-fetch` (a small wrapper over native `fetch`, no heavy codegen runtime) — this avoids hand-mirroring Java DTOs in TypeScript, the main type-drift risk in a solo-maintained cross-language stack. Generated output is committed to the repo (not gitignored); CI additionally re-runs codegen and fails the build if the committed output would differ from a fresh generation, catching drift automatically. (Wiring that CI drift check into `frontend-ci.yml` is a follow-up — it needs the relevant backend service running to generate against, not yet added as of this decision.) Each generated client is wrapped with a thin layer that pulls the access token from the Auth.js session (`auth()`) and sets it as the `Authorization: Bearer` header. Reads default to `cache: 'no-store'` rather than Next.js's default cached `fetch`, since for live device/system state staleness is worse than the extra backend round-trip. On a failed backend call (5xx, network error, timeout), a Next.js `error.tsx` boundary per route segment catches it and shows a friendly message with a manual retry action — no automatic retries for now.
- **Real-time update mechanism**: Server-Sent Events, relayed through the Next.js server — not WebSockets, and not direct browser-to-backend connections. Each backend service exposes its own SSE endpoint for domain state-change events (sensor readings for `greenhouse`, on/off state for `lighting`) via Spring MVC's `SseEmitter` (no new dependency, no move to WebFlux); how each service gets fresh data internally (ESP32 polling, tinytuya status) stays that service's own concern per backend-spec.md. The browser never connects to a backend service directly: a Route Handler per domain (`app/api/stream/greenhouse/route.ts`, `app/api/stream/lighting/route.ts`, mirroring the existing per-domain route structure) checks the session cookie, attaches the Bearer token server-side, opens the SSE connection to the backend service, and pipes it to the browser, which just uses a plain same-origin `EventSource`. This isn't only for consistency with the API client strategy above — browser `EventSource` can't set an `Authorization` header at all, so a direct browser→backend stream would need the token in a query string (logged, leaked); relaying through Next.js avoids that outright. SSE was chosen over WebSockets because commands already go through normal request/response Server Actions, so there's no need for client→server push on this channel — SSE is plain HTTP, gets free reconnection from `EventSource`, and avoids WebSockets' ingress/protocol-upgrade complexity for no real benefit here. SSE keep-alive (periodic comment lines) runs every 15s to hold connections open through proxies — infra-spec.md's Gateway needs a matching idle-timeout comfortably above that (e.g. 60s+). Each domain page reconciles its server-rendered initial state (from the no-store `fetch` reads) with incoming SSE events via one shared client hook per domain (e.g. `useDeviceState`), rather than each domain reimplementing that merge logic independently. Actuator state changes pushed this way are expected to reach the UI within 2 seconds of the change occurring, satisfying `requirements/manage-greenhouse-devices` FR6's state-reflection criterion.
  - **v1 exception — greenhouse**: per [backend-spec.md](backend-spec.md)'s ESP32 v1 contract, the `greenhouse` service has no background polling loop on the live-read path and no actuator state read-back, so there's nothing yet for an SSE endpoint to push. The greenhouse domain therefore skips the SSE route handler for v1: a Client Component polls a normal Route Handler on a fixed interval (proposed default 5s, matching the DHT22's own minimum sampling interval) to refresh the DHT reading and last-known actuator state, with the backend calling the ESP32 fresh on each of those requests. This is a deliberate, temporary deviation — the SSE relay above remains the target architecture for greenhouse once the ESP32 firmware update (deferred, see backend-spec.md's Open items) adds real state read-back.
- **Styling and component primitives**: Tailwind CSS — it's Next.js's own scaffolding default, has zero runtime cost (compiles to static CSS, nothing executes per-request), and is the only mainstream styling option with clean Server Components compatibility (CSS-in-JS libraries like styled-components/Emotion still need `"use client"` workarounds and lack first-class RSC support). Paired with shadcn/ui (Radix UI underneath) for interactive primitives (switches, sliders, dialogs, forms) — components are copy-pasted into the repo rather than pulled in as an opaque dependency, keeping full control consistent with this spec's general preference against heavyweight abstractions, while still getting accessibility (ARIA, keyboard nav) for free from Radix. Configured with the "new-york" style preset, neutral base color, and `lucide-react` icons. Forms use `react-hook-form` for form state plus Zod for validation, pairing naturally with the OpenAPI-generated types.
- **Testing frameworks**: Vitest over Jest for unit/component tests — esbuild/Vite-based so substantially faster, native ESM (no config fighting with the App Router), and close enough to Jest's API to have no real learning-curve cost; it's also what Next.js's own docs now point to for new projects. React Testing Library sits alongside it, scoped to Client Components, hooks, and pure functions — Server Components don't have a stable isolated-rendering story for RTL today, so route-level behavior involving them is verified through Playwright instead. Playwright over Cypress for E2E — genuinely multi-browser, faster, native TypeScript, and supports multiple isolated browser contexts in one test, which matters for the Keycloak Authorization Code redirect leaving and re-entering the app. Unit/component tests always mock the API layer at the generated-OpenAPI-client boundary (not the Server Action that wraps it), keeping one consistent mocking boundary across all tests and leaving each Server Action's own wrapping logic exercised for real. Playwright E2E mocks each domain's backend service rather than running against the real `greenhouse`/`lighting` services or Postgres: Server Actions execute server-side inside the Next.js process, so browser-level request interception (`page.route()`) can't reach that outbound call, and standing up the full real backend stack just to drive UI-level E2E assertions isn't worth the cost for a solo-maintained project. The mechanism is a small standalone HTTP stub per domain (e.g. `e2e/mock-greenhouse-server.mjs`, zero extra dependencies), started via a second entry in Playwright's `webServer` array, with that domain's `*_SERVICE_URL` env var pointed at it for the Next.js dev server's env — the real Server Action/typed-client code path runs unmodified; only the upstream URL differs between e2e and production. Keycloak is the one exception: E2E does run a real (or lightweight containerized) Keycloak for the login/redirect flow specifically, since that's the actual point of Playwright's multi-context support — everything else in the domain flows stays mocked as described above.

## Known issues / local dev gotchas
- **Stale browser cache after a `next dev` restart**: if the underlying `next dev` (Turbopack) worker process restarts while a browser tab is already open against it — e.g. after a `git checkout`/branch switch or a batch of file edits in the working directory it's watching, or simply stopping and restarting the dev server — that already-open tab can keep serving JS chunks tied to the *previous* server instance. Symptoms: a stuck loading indicator, and/or controls (e.g. the greenhouse LED/fan switches) that render but are inert or perpetually disabled, with zero console errors and zero failed network requests, making it look like a real app/hydration bug. The diagnostic tell is that a private/incognito window (no cache) always loads cleanly — if that works but the normal tab doesn't, it's this, not application code. A plain refresh, and even a hard refresh (`Ctrl+Shift+R`), is **not** reliably sufficient: the browser can still reuse a cached response body via conditional (`304`) revalidation, and DevTools' "Disable Cache" toggle only affects requests made through the DevTools-instrumented network stack while the panel is attached — it does not purge the on-disk HTTP cache for already-cached resources. The reliable fix is a full cache clear for the site, then reload (Firefox: `Ctrl+Shift+Delete` → "Clear Recent History" → range "Last Hour" → only "Cache" checked → Clear Now; Chrome: DevTools → right-click the reload button → "Empty Cache and Hard Reload"). Restarting `next dev` itself does not fix an already-open tab's stale state — the fix is client-side, in the browser.
  - **Fixed at the config level**: `next.config.ts`'s `headers()` sends `Cache-Control: no-store, must-revalidate` for `/_next/static/:path*`, but only when `NODE_ENV !== "production"`. This stops the browser from having anything to conditionally revalidate in dev, so the stale-304 scenario above shouldn't recur going forward — the manual full-cache-clear steps above are the fallback if it's ever still seen. This is dev-only and deliberately does not touch production caching: `next build` output is content-hashed (each bundle's filename encodes its content), so the same URL never serves different content across builds the way a live Turbopack dev server can across restarts — see infra-spec.md's "Open / not yet defined" list for the different, real production risk this doesn't address (`ChunkLoadError` on a client mid-rollout).
